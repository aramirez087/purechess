## Key Learnings (added 2026-06-13 — purechess-improve S03 puzzle serving + puzzle Glicko)

- **Puzzle Glicko REUSES the game engine verbatim — no extraction was needed.** `updateRating(player, games)` in `apps/api/src/ratings/glicko2.ts` is ALREADY a pure, exported, paper-exact function (it was never private to `RatingsService`). `PuzzleRatingService.applyResult` (`apps/api/src/puzzles/puzzle-rating.service.ts`) imports it directly and models one puzzle as one Glicko game: `opponent = { rating: Puzzle.rating, ratingDeviation: Puzzle.ratingDeviation }`, `score = solved ? 1 : 0`. **Import path for S11 chart / S04 solve loop: `import { updateRating, DEFAULT_RATING } from '../ratings/glicko2'`.** Game ratings are provably unchanged — the Glicko paper worked-example vector (1500/200/0.06 vs three opponents → 1464.06/151.52/0.05999) is pinned in BOTH `test/ratings/glicko2.spec.ts` and `test/puzzles/puzzle-rating.service.spec.ts`.

Defaults 1500/350/0.06 (match schema). Clamp a zero-RD puzzle opponent to a floor (60) so a sparse-seed `ratingDeviation=0` doesn't collapse the variance term (NaN). `get()` returns the defaults (no DB row) for an unrated user.
- **getNext selection ladder** (`apps/api/src/puzzles/puzzle-serving.service.ts`): target rating = explicit arg → user's `PuzzleRating.rating` → 1500. Built with composable `Prisma.sql` fragments + `Prisma.join(conds,' AND ')` so the WHERE is fully parameterized regardless of active filters; `ORDER BY random() LIMIT 1`. Theme filter = `p.themes @> ARRAY[$theme]::text[]` (GIN). **Test trick:** `$queryRaw` receives a `Prisma.Sql` object exposing `.sql` (text with `?` placeholders) + `.values` (bound params) — assert on those for mutation-proof SQL tests (no DB needed).

## Key Learnings (added 2026-06-13 — purechess-improve S12 insights & weakness engine)

- **Insights = PURE detectors + a thin orchestrator.** `apps/api/src/insights/weakness-detectors.ts` holds 5 pure fns, each `(already-aggregated input) → WeaknessDto | null`, independently unit-tested (40 cases, the trust-critical suite). `insights.service.ts` does ONLY the DB reads (one gatherer per detector, reusing existing services) + ranking. Keep the reasoning in the pure layer; never inline a detector's logic into the service.
- **Every detector is CONSERVATIVE = volume floor AND clear-gap.** Exported thresholds: `THEME_MIN_ATTEMPTS=15` + `THEME_WEAK_ACCURACY=0.65`; `RECURRING_MISTAKE_MIN=3`; `OPENING_LEAK_MIN_LAPSES=3`; `ENDGAME_GAP_MIN_UNSOLVED=2`. A detector returns `null` (not a low-severity card) below the floor — silence beats a false weakness. Test each on positive + noise + insufficient-data + empty.
- **`timeManagement` is the most conservative and needs a BASELINE comparison.** Fast-blunder signal fires only when fast-move (<3000ms) blunder rate ≥0.25 AND ≥2× the user's OVERALL blunder baseline — without the ×2 guard a uniformly-blundery player is misread as a time problem. Flag-loss signal needs ≥10 decisive games AND ≥0.25 flag rate AND ≥2 absolute flags (one unlucky flag never fires). A bogus time nudge is the fastest trust-killer.
- **Ranking = `severity × impact`, deduped by `kind`.** `rankWeaknesses` (pure, exported from insights.service): drop nulls → dedupe by `kind` (keep strongest per domain) → sort by `score = severity × (estimatedEloUpside ?? 10)` desc → cap 5. `score()` also pure+exported. `game-mistake` and `theme` are DISTINCT kinds (don't collapse) so a real-game motif and a puzzle gap can co-appear.
- **Theme commonness weighting: a weak `fork` outranks a weaker `enPassant`.** `themeCommonness(slug)` (0..1 map, unmapped→0.6 mid) multiplies the accuracy gap so common/high-leverage motifs win when picking the worst theme AND raise `estimatedEloUpside`. The S01 stub's `WeaknessDto.area` enum has no `game-mistake`, so I added `kind?: WeaknessKind` (optional) alongside `area` rather than touch the frozen-shape enum — `area` stays back-compat (game-mistake collapses to 'theme'), `kind` carries the real domain.
- **Time-management read is fully read-only, NO live engine.** Reads `Move.moveTimeMs` for speed + the already-persisted `GameMistake.cpLoss>=150` rows (keyed `gameId:ply`) as the per-move blunder marker — the FEN trail is NOT re-evaluated. Flag losses = `Game.result` + `resultReason==='timeout'`. Capped at the last 30 completed games.
- **InsightsModule imports the AGGREGATOR modules to inject their exported services** — `PuzzlesModule` (PuzzleServingService + GameMistakeService), `EndgamesModule` (EndgamesService), `RepertoireModule` (RepertoireReviewService), `AuthModule` (guard). Redis via `@Inject('REDIS_CLIENT')` from the global RedisModule. Cache key `insights:<userId>`, `setex` TTL 900s; read/write failures logged + degrade to fresh compute (never throw into the request path — same pattern as TablebaseService).
- **`InsightsController` is `@Controller('train')`** (route `GET /train/insights`), a NEW controller (not on a puzzles controller) — the insights surface lives under `/train`, sibling to the future S13 hub. ## Key Learnings (added 2026-06-13 — purechess-improve S14 adaptive difficulty + coach feedback)

- **Adaptive difficulty is a PURE damped proportional controller** (`apps/api/src/puzzles/adaptive-selector.ts`). `nextDifficulty(recent, currentTarget, anchor=currentTarget)`: `error = recentSuccessRate(last 10) - 0.80`; `step = error*GAIN(800)` + speed/fail nudges; `step` clamped to ±`MAX_STEP(60)`; result clamped to `anchor ± BAND(300)`. **EMPTY `recent` => returns `Math.round(currentTarget)` UNCHANGED** — this is the backward-compat no-op. Speed bonus (+25) on a trailing run of ≥3 fast solves (<8000ms); fail nudge (−20 per fail past the 2nd) on a trailing fail streak — both react to the tail of the window faster than the average would. Steers toward the 75-85% productive-struggle band (setpoint 0.80). `interleaveThemes(weakThemes, recentThemes)`: bias to the WEAKEST theme but never serve the same theme >2 in a row (`MAX_SAME_THEME_RUN=2`) — **spacing beats blocking** for retention; returns `null` (unfiltered) when no weak themes OR when the only weak theme is the blocked one.
- **First-run calibration window = `< CALIBRATION_ATTEMPTS (8)` attempts.** While calibrating: wider search band (`calibrationBand` → `CALIBRATION_BAND_HALF=350` vs steady `STEADY_BAND_HALF=150` which equals the S03 first tier), larger gain (`CALIBRATION_GAIN=1400`) and larger step cap (`CALIBRATION_MAX_STEP=140`) so a new user's level is found in a few solves. Post-calibration it settles to the damped steady law.

An explicit `theme` arg short-circuits the interleave (caller wins). **Opt-in was chosen over auto-detect specifically so adding a `puzzleAttempt.findMany` call couldn't disturb the S03 mocks.**

## Key Learnings (added 2026-06-13 — PGN shape round-trip)

- **AnalysisNode now has `shapes?: BoardShape[]`** (analysis-tree.ts) — user-drawn arrows/circles + shapes parsed from `[%cal]`/`[%csl]` PGN directives. Optional, so buildTree/addMove/nodeAtPath ignore it.
- **`[%cal][%csl]` parse/export is symmetric.** pgn-parser `parseShapesFromComment(text)` strips `[%cal G{from}{to},...]` (arrows) + `[%csl G{sq},...]` (circles) from a `{ }` block, returns `{shapes, remaining}`; remaining prose becomes `node.comment` (undefined when empty). Color codes G/R/Y/B ↔ green/red/yellow/blue. pgn-export.ts `exportTreeToPgn(root, headers?)` is the inverse: `{ [%cal ...][%csl ...] text }` — **shapes first, then text**, comment block only when shapes OR text present.
- **pgn-export tree-walk gotcha (bug-472):** a node's `children[0]` is the mainline continuation, `children[1..]` are alternatives to `children[0]` — so variations must emit AFTER the `children[0]` move token, not after the node itself. The parser IGNORES move numbers (tokenizer skips `\d+\.*` / `\.+`), so round-trip correctness needs only structure+SAN+NAG+comment+shapes, not perfect numbering; numbers are emitted clean anyway (white always `N.`, black `N...` only after a variation/comment). Start side+fullmove read from `root.fen`.
- **Chessboard shape persistence:** `onShapesChange` (user edit → parent saves to node) + `externalShapes` (parent-driven restore on navigation, does NOT fire onShapesChange). analyze-board mutates `nodeAtPath(tree.root, tree.path).shapes` directly (same live-mutation pattern as addMove); the path-change re-render recomputes `restoredShapes`. ## Key Learnings (added 2026-06-10 — S02 realtime resilience)

- **socket.io reconnection opts live in `use-game-socket.ts`'s `io(...)` call** (now explicit: `reconnectionAttempts: Infinity`, `reconnectionDelay: 500`, `reconnectionDelayMax: 5000`, `randomizationFactor: 0.5`, `timeout: 10000`). Bounded exp backoff + jitter is socket.io's native model — we only made it explicit and lowered the warm floor.
- **Tab sleep kills timers WITHOUT a socket `disconnect`** → the `connect`-driven resync never fires and the board sits stale. Fix pattern: `window 'online'` + `document 'visibilitychange'→visible` listeners that call a `resync()` (nudge half-open socket via `socket.connect()`, re-emit idempotent `GameJoin`, then the caller's guarded `onResync`). 250ms throttle collapses the online+visibility double-fire. ALL resync paths must reuse `onResync` (→ `refetchState` → `isStaleState` merge) — never add a 2nd merge path.
- **Same tab-sleep problem hits `useLiveClock`**: the 200ms interval freezes when backgrounded, so the clock shows a frozen stale value and "drifts in" on wake. Fix: a `visibilitychange→visible`/`online` listener that forces ONE `setTick` so the next render recomputes against real `Date.now()` + offset. Formula unchanged (offset tests stay green).
- **Gateway ping tuning = `@WebSocketGateway` decorator options** (`pingInterval`/`pingTimeout`). No `IoAdapter` in `main.ts`, and the decorator options DO reach the engine — verified via the engine.io handshake (`curl '.../socket.io/?EIO=4&transport=polling'` returns `pingInterval`/`pingTimeout` in the open packet). Set 10s/10s → dead-peer detection ≤~20s vs the ~45s default (25s/20s). The metadata key is `'websockets:gateway_options'` (`Reflect.getMetadata` on the gateway class) — usable to lock the values in a unit spec.
- **Neon keepalive in `PrismaService`**: `SELECT 1` every 4 min (< Neon's 5-min autosuspend), gated on `NODE_ENV!=='test'`, `unref()`'d, `clearInterval` in `onModuleDestroy`. Defeats autosuspend (compute stays billed/warm) in exchange for never eating a 0.5–2s cold wake on the first move after idle. Local Postgres can't reproduce Neon autosuspend, so the benefit is argued, not locally measured.
- **Live reconnect (kill+restart API mid-game): board NEVER rewinds** (8/8 trials, ply+fen preserved) because the socket only triggers a REST resync through `isStaleState` — it never writes board state. Reconnect latency from API-up is 7/8 < 800ms (median ~640ms) with a tail outlier (~2.3s) caused purely by socket.io backoff-window alignment during the outage, not resync logic.

## Key Learnings (added 2026-06-11 — S03 board feel a11y)

- **SR narration: single Chessboard `role="status"` covers all game modes.** Every mode (PvP,
  computer, review, analyze) drives the board via the `position` prop. A `useRef`/`useEffect` pair
  diffs consecutive `position` values and calls `buildMoveAnnouncement` (lib/board/sr-announce.ts).
  Adding per-mode announcers is wrong — they would double-announce and miss mode switches.
- **`buildMoveAnnouncement` uses chess.js 4-field FEN prefix matching.** Enumerate moves from
  prevFen, apply each via `new Chess(prevFen).move({from,to,promotion})`, compare first 4 FEN
  fields against nextFen. Flags `k`/`q` = castling; `test.isCheckmate()`, `inCheck()`, etc. for
  suffixes. Returns null for same FEN or try/catch (invalid FEN). Does NOT parse SAN strings.
- **Pointer capture after drag threshold, not on pointerdown.** `e.currentTarget.setPointerCapture`
  must be called inside the `if (!isDragging.current && sqrt > DRAG_THRESHOLD)` branch in
  `use-drag.ts`. Capturing on pointerdown steals the click event and breaks tap-tap piece moves.
  Wrapped in try/catch (setPointerCapture unsupported on some legacy devices).
- **Focus ring on chessboard squares: use boxShadow sandwich, not Tailwind ring.** Tailwind
  `ring-2 ring-[--brass]` is invisible on bone (light) squares where the brass ring composites
  against near-identical fill. Instead: `boxShadow: 'inset 0 0 0 3px rgba(0,0,0,0.45), inset 0 0 0 2px hsl(41 56% 62%)'` (dark inner + brass outer) passes AA on both bone and mineral squares.
- **`aria-selected` on `gridcell` for keyboard selection state.** Correct ARIA for "this cell is
  chosen" inside a `role="grid"` widget. Set to `true` when `isSelected || isKeyboardFocus`, else
  `undefined` (omitted from DOM). Do NOT use a live region to announce focus moves — that would
  fire on every arrow key and be extremely chatty.
- **`useKeyboard` Space key = Enter.** Add `case ' ':` as a fallthrough before `case 'Enter':`.
  Space is the natural "activate" key for keyboard-only users; without it, a keyboard player who
  doesn't know to press Enter cannot play. Also call `e.preventDefault()` on Space (prevents page
  scroll).
- **Orientation-aware initial focus in `useKeyboard`.** `useState<Square>(() => orientation ===
  'white' ? 'e2' : 'e7')` + `useEffect(() => { setFocusSquare(…) }, [orientation])`. On mount,
  the cursor starts at the side's active pawn rank (board-flip in review mode resets it).

## Key Learnings (added 2026-06-13 — daily puzzle /puzzles)

- **Lichess puzzle `initialPly` = 0-based index of the game's LAST move (the setup move); the puzzle position is the FEN AFTER it.** So `puzzleFen = plies[initialPly].fenAfter`, `setupMove = plies[initialPly]`, `preSetupFen = plies[initialPly-1].fenAfter` (or start when 0). The pgn ends exactly at the puzzle position. The side to move there is the solver, and `solution[0]` is the solver's move. ## Decision Log — 2026-06-13: S07 mistakes-from-your-games

- **Mistake review is a BACKLOG, not a PuzzleReview card.** `PuzzleReview.puzzleId` is an FK to `Puzzle` (confirmed schema.prisma:388), and `GameMistake` has NO spaced-repetition columns (no dueAt/intervalDays/easeFactor). So a `mistake:<id>` row in PuzzleReview would violate the FK, and a mistake cannot carry a true SM-2 schedule under the frozen schema. - **Anti-spoof FEN re-derivation uses the stored `Move` rows, not a chess.js replay.** For a claimed mistake at ply P: the position BEFORE = `Move[ply=P-1].fenAfterMove` (or `game.startingFen`/standard at P=1); the played move = `Move[ply=P].uci`. The service compares the client's claimed `fen` (placement/turn/castling/ep only — clocks ignored via `.split(' ').slice(0,4)`) and `playedUci` against these and SKIPS any mismatch. It also enforces own-side-only via ply parity (`ply % 2 === 1` ⇒ White) against the user's game side, and `cpLoss >= MISTAKE_CP_THRESHOLD` (150). One bad candidate is skipped, never throws — legitimate rows in the same batch still persist. The STORED fen is the server-derived one, never the client's claim.
- **Threshold = 150 cp** (`MISTAKE_CP_THRESHOLD`, exported from game-mistake.service.ts; mirrored client-side as `CAPTURE_CP_THRESHOLD` in use-mistake-capture.ts). Matches the classifier's mistake/blunder bands. The client filters on `class ∈ {mistake,blunder}` AND capped `cpl >= 150`.

- **bug-601: Postgres has NO `max(boolean)` aggregate.** `prisma.X.groupBy({_max:{boolColumn:true}})` / `.aggregate({_max:{boolColumn}})` throws `function max(boolean) does not exist` (42883) against a REAL DB — but mocked unit specs never catch it (they return a fake `_max`). `EndgamesService.list()/getBySlug()` did this on `EndgameAttempt.succeeded`, 500-ing `/api/train/plan` (insights→endgames gaps). Fix: count succeeded attempts instead (`groupBy` with `where:{succeeded:true}` for `list`, two `count()`s for `getBySlug`); `solved = successCount > 0`. **Lesson: a boolean "did any row pass?" rollup = COUNT with a where-filter, never `_max` on the bool.** E2E against a real DB surfaced it; pure-mock unit tests can't.
- **E2E board interaction flakes under `next dev` (not in prod build).** The Chessboard layers a pointer-drag engine over a click-to-move handler; under on-demand route compilation + hydration the FIRST click after a route compile can be swallowed. Mitigations: pre-warm routes in `beforeAll`, wait for the player-phase prompt (`Find/Best move for White/Black.`) before clicking, hydration settles, `expect(async()=>{...}).toPass()` retry wrappers. Run E2E against a PRODUCTION build (`next build && next start`) for stable clicks.
