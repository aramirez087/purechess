## 0. Ground-truth findings (verified this session)

These override two assumptions carried in the operator rules / Session-01 handoff.
Read before coding.

1. **No column named `engineStateSnapshot` exists.** The actual column is
   `Game.engineState Json?` (migration `20260607140000_computer_game_engine_state`
   adds `engineState` + `lastComputerMove`; the SQL literally adds a column called
   `engineState`). The serialized `EngineState` (fen, fenHistory, moves, clock,
   `pendingDrawOfferBy`, status, result) IS the snapshot. **Reuse `engineState`.**
   Do not add/rename a column.

2. **NO Prisma migration is needed for this session.** Verified against
   `apps/api/prisma/schema.prisma`:
   - `enum GameStatus { pending active completed aborted invite_pending }` — `aborted`
     already exists. Writing `status: "aborted"` works (existing code already writes
     `"active"`/`"completed"` string literals against this enum).
   - `enum GameResultReason { … stalemate insufficient_material threefold_repetition
     fifty_move_rule draw_agreement abandonment }` — every termination Session-02 needs
     is present.
   - Draw-offer state does **not** need a `drawOfferedBy` column: `EngineState`
     already has `pendingDrawOfferBy: Color | null`, which is serialized into
     `engineState` and survives round-trips. Derive `drawOffered` / `drawOfferedBy`
     from it.
   - Therefore Session-01's "owns: migration for `status='aborted'` + `drawOfferedBy`"
     is a no-op. **Document this in the handoff so downstream doesn't expect a new
     column.** (Only add a migration if a reviewer insists on a denormalized
     draw-offer column — not required for correctness.)

3. **No computer-games controller/service spec is missing-but-named.** Existing specs:
   `apps/api/test/computer-games/computer-games.service.spec.ts` (mocks
   `PrismaService, EngineService, PosthogService` — pattern to copy) and
   `stockfish.service.spec.ts`. The service spec already contains a flag-fall/bug-005
   test to extend.

4. **`submitMove` currently force-disables the clock**: line ~224 sets
   `engineState.clock.lastTickAt = nowMs` before every `applyMove`, so `isTimeout`
   sees ~0 elapsed and never fires (computer games presented as `∞`). Clock-awareness
   = gate this reset on `timeControlSeconds <= 0` (untimed) and let `applyMove` tick
   real elapsed when `timeControlSeconds > 0`.

5. **bug-005 guard already present** in `submitMove` (`if (state.moves.length ===
   engineState.moves.length)` early-return). Keep it; the new timed path flows through
   the same `applyMove`, so the guard already covers flag-fall. Just assert it in tests.

6. **`EngineMove` carries `by: Color`** (see `game-state.ts` line 95) and `fenAfter`,
   `clockAfterMs`, `ply`. This is enough to reconstruct state on truncation without
   replaying moves — used by the takeback/rewind helper below.

7. Engine internals (`makeClock`, `serializeClock`, `fenPosition`) live in the
   coverage-gated `chess/engine/` dir and are **not** re-exported by `EngineService`.
   To avoid touching the gated dir, create-from-fen and the truncate helper build the
   `SerializableEngineState` literal directly (position-key = first 4 FEN fields;
   clock = plain `{whiteMs,blackMs,lastTickAt,incrementMs}` numbers).

## 2. Design decisions + rationale

**D1. Separate `ComputerGameActionsService`, keep `submitMove`/`createGame`/`getGame`
in `ComputerGamesService`.** Matches the task. `rematch` calls
`ComputerGamesService.createGame`; inject the core service into the actions service
(one-way dep, no cycle).

**D2. Truncation helper (takeback + rewind share it).** Pure function
`truncateToPly(serialized, targetPly, game, nowMs)`:
- `moves = serialized.moves.slice(0, targetPly)`
- `fen = targetPly > 0 ? moves[targetPly-1].fenAfter : STARTING_FEN`
  (or `game.startingFen` for create-from-fen games)
- `fenHistory = serialized.fenHistory.slice(0, targetPly + 1)`
- `clock`: whiteMs = last `by==='w'` move ≤ target's `clockAfterMs`, else
  `timeControlSeconds*1000`; blackMs symmetric; `lastTickAt = nowMs`;
  `incrementMs` unchanged.
- `status = 'active'`, `result = null`, `resultReason = null`,
  `pendingDrawOfferBy = null` (truncation clears a stale offer).
Reconstructs from per-move data — **no move replay**, so no chess.js re-execution and
no engine-dir changes. Re-serialize and persist.

**D3. Respect `(gameId, ply)` UNIQUE — DELETE, never re-insert.** Both takeback and
rewind run in a `prisma.$transaction`:
`tx.move.deleteMany({ where: { gameId, ply: { gt: targetPly } } })` then
`tx.game.update({ engineState, finalFen, pgn, lastComputerMove, status:'active',
result:null, resultReason:null, endedAt:null })`. PGN rebuilt via
`engine.buildPgn(moves, headers)`. `lastComputerMove` = uci of last move with
`by === computerColorChar`, else `null`.

**D4. Takeback semantics.** `plies: 1 | 2`. `targetPly = moves.length - plies`.
Reject `targetPly < 0` → `BadRequestException('Nothing to take back')` (covers the
takeback-with-no-moves case). `plies:2` removes human move + bot reply; `plies:1`
removes only the trailing ply. (No turn-parity enforcement beyond bounds — client
sends the correct count per the DTO contract; server only guarantees row integrity.)

**D5. Rewind bounds.** `RewindToPlyDto.ply` = target length. Reject `ply < 0` or
`ply > moves.length` → `BadRequestException('Ply out of range')`. `ply === moves.length`
is a no-op (allowed, returns current state) — or reject as "nothing to rewind";
choose reject for a tighter contract and a clean bounds test.

**D6. Abort = zero human moves only.** `humanColorChar = computerColor==='white'?'b':'w'`.
`humanMoves = moves.filter(m => m.by === humanColorChar).length`. If `> 0` →
`BadRequestException('Cannot abort after a move')`. Else set `status:'aborted'`,
`result:null`, `resultReason:null` (aborted games carry no result; unrated anyway →
no rating impact, consistent with `isRated:false`), `endedAt: now`. `abortable`
on the DTO = `status==='active' && humanMoves===0`. (Session-01 mentioned
`GameTermination.Abandonment`; we leave `result/resultReason` null since `aborted`
is a distinct status, not a `completed` result. Documented in handoff.)

**D7. Draw actions** (`DrawActionDto.action`):
- `offer` → `engineState.pendingDrawOfferBy = humanColor`; persist; DTO
  `drawOffered:true, drawOfferedBy:humanColor`. Game stays active.
- `accept` → require `pendingDrawOfferBy != null` (else `BadRequestException`);
  complete: `status:'completed'`, `result: GameResult.Draw`,
  `resultReason: GameTermination.DrawAgreement` (→ `draw` / `draw_agreement`),
  clear offer, `endedAt`.
- `decline` → clear `pendingDrawOfferBy`; persist; game stays active.
- `claim` → `engine.detectResult(state, now)`; if it returns a `Draw` reason
  (stalemate / threefold_repetition / fifty_move_rule / insufficient_material) →
  complete with that `result`/`reason`; else `BadRequestException('No draw to claim')`.
  Uses the existing engine `detectResult` (reads position + fenHistory + clock).
- Cast to Prisma enums exactly as existing code does
  (`reason as unknown as GameResultReason`, `result as unknown as PrismaGameResult`).

**D8. Rematch.** Load original game (ownership + `isVsComputer` checks). Build
`CreateComputerGameDto` from it: `level = computerLevel`,
`timeControlSeconds`, `incrementSeconds`, `color =` original human color
(`computerColor==='white'?'black':'white'`), flipped if `swapColors`. Delegate to
`ComputerGamesService.createGame(userId, dto)`. Returns **201**.

**D9. Create-from-fen.** In `ComputerGamesService` (not actions — it's a creation op
sibling to `createGame`). Validate `new Chess(fen)` (throws → 400). Create `Game`
(`startingFen=finalFen=fen`, `computerColor` from `color`, `isVsComputer:true`,
`isRated:false`). Build `SerializableEngineState` literal directly
(`moves:[]`, `fenHistory:[fen.split(' ').slice(0,4).join(' ')]`, clock numbers,
`status:'active'`). Persist. Returns **201**.

**D10. Clock-aware `submitMove`.** Gate the existing tick-reset on timing:
```
if (game.timeControlSeconds > 0) {
  // timed: keep persisted lastTickAt so applyMove ticks real elapsed,
  // applies increment, and isTimeout/flag-fall can fire
} else {
  engineState.clock = { ...engineState.clock, lastTickAt: BigInt(nowMs) }; // untimed
}
```
`applyMove` already does tick+increment+timeout (`game-state.ts`). The bug-005 guard
already persists the flag-fall result without writing a `Move` row. Populate `clock`
(from serialized `engineState.clock`) on every returned DTO. No engine changes.

**D11. DTO population.** Add a single `computeExtras(serialized, game)` returning
`{ clock, drawOffered, drawOfferedBy, abortable }`; thread into `toStateDto`.
`getGame` also returns these so the client can render offer/abort affordances on load.

**D12. Authoritative + validation.** Every action: load game →
`NotFoundException` if absent → `BadRequestException` if `!isVsComputer` →
`ForbiddenException` if user is neither player → state guards (active/offer/bounds).
Mirrors existing `submitMove`/`getGame` guard order. Server is sole source of legality.

## 3. Risks & mitigations

| Risk | Mitigation |
|---|---|
| `(gameId, ply)` collision on takeback/rewind | `deleteMany ply > targetPly` inside the same `$transaction` as the game update (D3). Test asserts deleted rows + re-submittable next ply. |
| bug-005 flag-fall 500 in timed path | Existing `state.moves.length === engineState.moves.length` guard covers it; add an explicit timed flag-fall test. |
| Clock reconstruction wrong after truncate | Derive each side's ms from its last surviving move's `clockAfterMs`; fall back to `timeControlSeconds*1000`; reset `lastTickAt=now`. Unit-test a 3-ply truncate. |
| Touching `chess/engine/**` trips the 90/90/85 gate | All new logic in ungated `computer-games` services; build serialized literals instead of calling gated helpers (D2/D9). |
| Changing untimed behavior breaks existing `∞` games | Gate timing on `timeControlSeconds > 0`; untimed path is byte-identical to today. |
| Reopening a `completed` game via takeback/rewind | Allow takeback/rewind from `completed` (undo a checkmate/timeout): set `status:'active'`, clear `result/resultReason/endedAt`. Block abort/draw on non-active games. |
| Invalid FEN → 500 | `try { new Chess(fen) } catch → BadRequestException`. |
| `pnpm -r lint` can't resolve eslint bin (fresh worktree, per cerebrum) | Run `node node_modules/.pnpm/eslint@8.57.1/.../eslint.js <files>`; note in handoff. |
| `@prisma/client` ungenerated → phantom typecheck errors | Run `pnpm --filter @purechess/api db:generate` before typecheck (cerebrum learning). |
| web typecheck red on pre-existing errors | Out of scope; do not fix; note in handoff (already flagged Session 01). |

## 4. Route table (controller) — handoff source of truth

| Method | Route | Body DTO | Returns | Status | Service call |
|---|---|---|---|---|---|
| POST | `/computer-games/:id/takeback` | `TakebackDto {plies:1\|2}` | `ComputerGameStateDto` | 200 | `actions.takeback` |
| POST | `/computer-games/:id/rewind`   | `RewindToPlyDto {ply}` | `ComputerGameStateDto` | 200 | `actions.rewind` |
| POST | `/computer-games/:id/abort`    | `AbortDto {reason?}` | `ComputerGameStateDto` | 200 | `actions.abort` |
| POST | `/computer-games/:id/draw`     | `DrawActionDto {action}` | `ComputerGameStateDto` | 200 | `actions.draw` |
| POST | `/computer-games/:id/rematch`  | `RematchDto {swapColors?}` | `ComputerGameStateDto` | 201 | `actions.rematch` |
| POST | `/computer-games/from-fen`     | `CreateFromFenDto` | `ComputerGameStateDto` | 201 | `service.createGameFromFen` |

`@HttpCode(200)` on takeback/rewind/abort/draw; `@HttpCode(201)` on rematch/from-fen.
Note: `from-fen` is a collection route — declare it **before** `:id` param routes is
not required (distinct path segment), but place it near `createGame` for clarity.

Error codes: 400 (bad state/bounds/FEN/no-offer/no-claim), 403 (not your game),
404 (game not found).

## 5. Test plan (`computer-game-actions.service.spec.ts` + service spec additions)

Mock `PrismaService` (`game.findUnique/create/update`, `move.create/deleteMany`,
`$transaction` running the callback), `EngineService`
(`fromSerializable/toSerializable/detectResult/buildPgn`), `PosthogService`.
Copy the `emptyEngineState` / serialized-state fixtures from the existing service spec.

Cases (each new path):
- **takeback** plies=1 (removes trailing ply, deletes `ply>target`), plies=2
  (human+bot), **takeback-with-no-moves → BadRequest**, takeback from `completed`
  reopens to `active`.
- **rewind** to mid-ply (deletes rows, restores fen/clock), `ply=0` (back to start),
  **out-of-range ply (<0 and >len) → BadRequest** (rewind bounds).
- **abort** before any human move → status `aborted`; **abort-after-move → BadRequest**.
- **draw**: offer sets `pendingDrawOfferBy`; accept-without-offer → BadRequest;
  accept-with-offer → draw/draw_agreement; decline clears offer; claim with a
  detectable threefold/fifty/stalemate → completes; claim with none → BadRequest.
- **rematch**: creates new game with same settings; `swapColors` flips human color;
  returns new gameId.
- **create-from-fen**: valid FEN → active game at that fen; invalid FEN → BadRequest.
- **clock-aware submitMove** (service spec): timed game ticks + applies increment and
  populates `clock` on DTO; **timed flag-fall** routes through the bug-005 guard
  (no `move.create`, status `completed`, reason timeout, **no 500/no ply collision**);
  untimed game unchanged.
- Common guards per action: NotFound, not-vs-computer, Forbidden.

Coverage: new logic is outside `chess/engine/` so it doesn't affect the engine gate;
the gate must still report ≥90/90/85 (no engine files touched) — verify it stays green.
