## Design Decisions and Rationale

### 1. Extend `Game` model rather than new `ComputerGame` model

**Decision:** Add three nullable fields to the existing `Game` model.

**Rationale:**
- Existing schema already made `whitePlayerId` / `blackPlayerId` nullable in migration `20260607120000_nullable_player_ids` â€” anticipating exactly this case.
- Reuses all existing game logic: moves table, ratings, PGN, FEN, timers, reports, audit log.
- Keeps query surface uniform; a `WHERE isVsComputer = true` filter suffices to partition the two game types.
- A separate `ComputerGame` model would duplicate ~30 columns and force two code paths everywhere.

**Fields added:**
```prisma
isVsComputer  Boolean  @default(false)
computerLevel Int?     // 1â€“8 â†’ UCI skill 0,3,5,8,11,14,17,20
computerColor String?  // 'white' | 'black'
```

Migration is additive: `isVsComputer` has a default so existing rows get `false`; `computerLevel` and `computerColor` are nullable so no backfill needed.

### 2. `stockfish` npm package (JS/WASM build)

**Decision:** `npm install stockfish` in `apps/api`.

**Rationale:**
- Works in any Node.js deployment without a system binary.
- Eliminates `child_process.spawn` and platform-specific PATH issues.
- Package exports a self-contained WASM + JS wrapper; UCI communication via message-passing.
- Session 02 will wire up the UCI bridge in `stockfish.service.ts`.

### 3. DTO types as plain TypeScript interfaces (no class-validator decorators yet)

**Decision:** `computer-game.dto.ts` exports TypeScript `interface` types, not NestJS `class` DTOs.

**Rationale:**
- Session 01 is scaffold-only; validation decorators are applied in session 02 when the controller methods are implemented.
- Shared package has no class-validator dependency and none should be added there.
- Type aliases with union literals (`'white' | 'black' | 'random'`) give compile-time safety without runtime overhead.

### 4. Empty NestJS module pattern

**Decision:** All four files in `apps/api/src/computer-games/` are minimal stubs with no injected dependencies.

**Rationale:**
- Avoids circular imports or missing provider errors at build time.
- Lets session 02 freely add constructor params without merge conflicts.
- `ComputerGamesModule` registered in `app.module.ts` proves the module wires cleanly before any logic lands.

### 5. `stockfish.service.ts` carries a single comment

Per session instructions, one comment: `// UCI bridge â€” filled in session 02`. No other comments anywhere.

## Risks and Mitigations

| Risk | Likelihood | Mitigation |
|------|-----------|------------|
| `prisma migrate dev` fails if DB not running locally | Medium | Plan notes that migration SQL file itself is the deliverable; CI runs migrations against a test DB. Developer must have local PG running. |
| `stockfish` npm package version ships WASM that Node.js version can't load | Low | Verify import resolves at `npm install` time; actual WASM load tested in session 02. |
| `isVsComputer Boolean @default(false)` causes lock contention on large `Game` table | Low | Migration adds column with DEFAULT; Postgres 11+ does this without a full table rewrite. |
| Naming collision: `ComputerColor` enum vs freeform `String?` | Low | Keeps `computerColor` as `String?` (not a Prisma enum) â€” avoids a new enum migration and matches the DTO union type at runtime. |
| `packages/shared` ESM exports break if `.js` extension omitted in new export | Low | Mirror existing pattern: `export * from './dto/computer-game.dto.js'` (`.js` extension required for ESM node resolution). |
