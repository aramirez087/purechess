## Design Decisions

### 1. Schema Migration (necessary deviation from additive-only rule)

**Problem**: `Move.userId String` is NOT NULL. Computer moves have no userId. Session 02 requires computer moves recorded as Move rows.

**Decision**: Make `Move.userId String?` (nullable) and add `Move.isComputer Boolean @default(false)`. This is technically altering an existing column — a deliberate exception to the additive-only rule, justified because:
- Existing rows are unaffected (all have non-null userId today)
- No FK constraint is violated (nullable FKs are valid in Prisma/Postgres)
- The alternative (sentinel user, or skipping Move rows) is worse for data integrity

**Additional fields on Game** (additive, nullable — within rules):
- `engineState Json?` — serialized EngineState for round-trip (needed to reconstruct clock + fenHistory for subsequent moves without replaying full history from Move rows)
- `lastComputerMove String?` — UCI notation of most recent computer move for fast DTO construction

**Migration SQL** (additive except the one necessary alteration):
```sql
ALTER TABLE "Game" ADD COLUMN "engineState"       JSONB;
ALTER TABLE "Game" ADD COLUMN "lastComputerMove"  TEXT;
ALTER TABLE "Move" ALTER COLUMN "userId"          DROP NOT NULL;
ALTER TABLE "Move" ADD COLUMN  "isComputer"       BOOLEAN NOT NULL DEFAULT FALSE;
```

### 2. StockfishService: Per-Call Instance Pattern

**Decision**: Spawn one Stockfish instance per `getBestMove()` call (not a singleton). Rationale: avoids state leaks between concurrent calls; simpler lifecycle; memory reclaimed immediately after each call.

**stockfish npm API** (v18 JS/WASM, CommonJS interop):
```typescript
// The package exports a factory: stockfish() returns an engine with postMessage / onmessage
const Stockfish = require('stockfish');
const engine = Stockfish();
engine.onmessage = (line: string) => { /* parse UCI output */ };
engine.postMessage('uci');
```

**ESM risk**: `stockfish` v18 may be pure ESM. If `require()` fails, use dynamic `import()` inside the async method. Plan includes fallback import pattern.

**UCI command sequence**:
```
uci
(wait for uciok)
setoption name Skill Level value <N>
isready
(wait for readyok)
position fen <fen>
go movetime <ms>
(wait for bestmove line)
quit
```

**Timeout**: 5000ms via `Promise.race` with a `setTimeout` rejection.

**Level mapping** (index = level - 1):
```typescript
const UCI_SKILL = [0, 3, 5, 8, 11, 14, 17, 20];
```

### 3. ComputerGamesService: Engine State Round-Trip

**Problem**: `EngineService.applyMove` needs a full `EngineState` including clock snapshot and fenHistory. On `submitMove`, we can't reconstruct this cheaply from Move rows alone (clock state is lost).

**Decision**: Serialize `EngineState` to `Game.engineState` (Json column) after every state change using `EngineService.toSerializable()` / `fromSerializable()`. Trade-off: ~2KB JSONB per game row; worth it for correctness.

### 4. Color Resolution

```typescript
function resolveColor(dto: CreateComputerGameDto): 'white' | 'black' {
  if (dto.color === 'random') return Math.random() < 0.5 ? 'white' : 'black';
  return dto.color;
}
```

If user color is `'white'`: `whiteUserId = userId`, `blackUserId = null`, `computerColor = 'black'`.  
If user color is `'black'`: `blackUserId = userId`, `whiteUserId = null`, `computerColor = 'white'`.

### 5. Turn Validation

Before applying user move in `submitMove`:
```typescript
const isWhiteTurn = engineState.position.turn() === 'w';
const userIsWhite = game.whiteUserId === userId;
if (isWhiteTurn !== userIsWhite) throw new BadRequestException('Not your turn');
```

### 6. Result Detection

After each move (user then computer), call `EngineService.detectResult(state, nowMs)`. If non-null, update `Game.status = 'completed'`, `Game.result`, `Game.resultReason`, `Game.endedAt`. Skip computer move if user move ended the game.

### 7. Move Row for Computer Moves

Computer moves written with `isComputer: true, userId: null`:
```typescript
await prisma.move.create({
  data: {
    gameId,
    moveNumber: Math.ceil(ply / 2),
    ply,
    userId: null,       // nullable after migration
    isComputer: true,
    san,
    uci,
    fenAfterMove: engineMove.fenAfter,
    clockAfterMoveMs: engineMove.clockAfterMs,
    moveTimeMs: engineMove.moveTimeMs,
  },
});
```

### 8. Auth — User Extraction

`SessionAuthGuard` sets `req.user`. Pattern from existing controllers (inferred):
```typescript
@Get(':id')
getGame(@Param('id') id: string, @Req() req: Request & { user: { id: string } }) {
  return this.service.getGame(id, req.user.id);
}
```

If `req.user` shape differs, adjust. Check other controllers if TS errors appear.

### 9. EngineService Clock for Computer Games

Computer "moves" have no real move time. Pass `nowMs` for both user and computer moves; the clock tick for computer is effectively instantaneous. For user turn, `nowMs = Date.now()`.

### 10. MoveIntent Shape

From `@purchess/shared`, `MoveIntent` is inferred as `{ from: string; to: string; promotion?: string }` based on UCI format `e2e4` → `{ from: 'e2', to: 'e4' }`. Parse UCI string:
```typescript
function uciToIntent(uci: string): MoveIntent {
  return {
    from: uci.slice(0, 2),
    to: uci.slice(2, 4),
    ...(uci.length === 5 ? { promotion: uci[4] } : {}),
  };
}
```

## Risks and Mitigations

| Risk | Likelihood | Mitigation |
|------|-----------|------------|
| `stockfish` npm is pure ESM, breaks `require()` in NestJS CommonJS | Medium | Use dynamic `import()` inside async method; fallback to `require` with try/catch |
| `stockfish()` factory API differs from expected (callback vs Promise) | Medium | Unit test mocks the factory; verify actual API with a quick smoke test separately |
| `MoveIntent` shape doesn't match UCI parse logic | Low | TypeScript will catch at compile; check `@purchess/shared` exports if TS errors appear |
| `engineState` JSON grows too large for Postgres | Low | EngineState is ~2KB; no concern for this scale |
| `Move.userId` alteration breaks Prisma foreign key in existing code | Low | Nullable FK is valid; existing code that queries moves will still work (userId can be null, check with optional chaining where needed) |
| Computer move returns `(none)` from stockfish (no legal moves) | Low | Check for `bestmove (none)` → treat as game over (stalemate/checkmate already detected) |
| Concurrent `submitMove` calls for same game | Low | No DB transaction locking yet; acceptable for MVP (add optimistic lock in later session if needed) |

2. **Update `schema.prisma`**: add `engineState Json?`, `lastComputerMove String?` to Game; make `Move.userId String?`; add `Move.isComputer Boolean @default(false)`.

3. **Create migration SQL** at `apps/api/prisma/migrations/20260607140000_computer_game_engine_state/migration.sql`.

4. **Implement `StockfishService`**:
   - Dynamic import of `stockfish` package
   - `getBestMove(fen, skillLevel, movetime?)` method

   - Assert `bestmove e2e4` parsed to `'e2e4'`

9. **Run quality gates** (build → lint → test). Fix any errors.
