## Pre-Work: Schema Fix

### Problem
`Move.userId` is `String` (non-nullable) with FK to `User`. Computer moves have no user — the computer is not a DB user. Storing computer moves in `Move` requires either a fake system user (fragile) or making the column nullable (clean).

### Decision: Make `Move.userId` nullable
Rationale: loosening NOT NULL is the safest possible migration — zero data change, zero risk to existing rows, no code paths break because the new nullable path is only hit for computer moves.

**`apps/api/prisma/schema.prisma` change:**
```prisma
model Move {
  ...
  userId           String?                     // was String
  ...
  user             User?   @relation(...)      // was User
}
```

**Migration SQL (`20260607140000_nullable_move_userid/migration.sql`):**
```sql
-- Make Move.userId nullable to support computer-generated moves
ALTER TABLE "Move" ALTER COLUMN "userId" DROP NOT NULL;
```

After schema change: run `npx prisma generate` inside `apps/api/`.

## Step 1: `StockfishService`

**File:** `apps/api/src/computer-games/stockfish.service.ts`

### `stockfish` npm package API (v16/v18 WASM)
The `stockfish` package exports a factory function. Each call creates a new engine instance with `postMessage(cmd)` for input and `addMessageListener(fn)` / `onmessage` for output.

```typescript
import stockfish from 'stockfish';
const engine = stockfish();               // new instance per call
engine.addMessageListener((line) => ...); // or engine.onmessage = (line) => ...
engine.postMessage('uci');
engine.postMessage('quit');
```

**If `addMessageListener` does not exist** (older API), fall back to `engine.onmessage`. The plan must handle both; the implementation will check at runtime.

### UCI Skill Mapping
```typescript
const UCI_SKILL = [0, 3, 5, 8, 11, 14, 17, 20]; // index = level - 1
```

### `getBestMove(fen, skillLevel, movetime?)` logic
1. Create engine instance via `stockfish()` factory
2. Attach message listener, collect lines
3. Send: `uci` → wait for `uciok`
4. Send: `setoption name Skill Level value <N>`
5. Send: `isready` → wait for `readyok`
6. Send: `position fen <fen>`
7. Send: `go movetime <ms>` (default 1000)
8. Wait for line matching `/^bestmove (\S+)/`
9. On match: send `quit`, resolve with the move string
10. Timeout after 5000ms: reject with Error

**Important:** The `go movetime` command may return `bestmove (none)` if the position is terminal (checkmate/stalemate). The service should handle this gracefully (return `null` or throw a specific error; caller must check terminal state before calling).

### Return shape
```typescript
async getBestMove(fen: string, skillLevel: number, movetime = 1000): Promise<string>
```

Throws `Error('Stockfish timeout')` if no bestmove within 5s.

## Step 2: `ComputerGamesService`

**File:** `apps/api/src/computer-games/computer-games.service.ts`

### Injections
```typescript
constructor(
  private readonly prisma: PrismaService,
  private readonly stockfish: StockfishService,
  private readonly engine: EngineService,
) {}
```

`PrismaService` is `@Global()` — no extra module import needed. `EngineService` comes via `ChessEngineModule` import in the module.

### `createGame(userId, dto)` flow

1. Resolve color:
   - `'white'` → user is white, computer is black
   - `'black'` → user is black, computer is white
   - `'random'` → `Math.random() < 0.5 ? 'white' : 'black'`

2. Compute `computerColor`: opposite of user color.

3. Create `Game` row:
   ```typescript
   prisma.game.create({
     data: {
       whiteUserId: userColor === 'white' ? userId : null,
       blackUserId: userColor === 'black' ? userId : null,
       timeControlSeconds: dto.timeControlSeconds,
       incrementSeconds: dto.incrementSeconds ?? 0,
       category: resolveCategory(dto.timeControlSeconds),
       isRated: false,         // vs-computer games are unrated
       isVsComputer: true,
       computerLevel: dto.level,
       computerColor,
       status: 'active',
       startedAt: new Date(),
     },
   })
   ```

4. **If computer is white** (plays first):
   - Call `stockfish.getBestMove(STARTING_FEN, dto.level)`
   - Apply move via engine to get SAN/UCI/fenAfter
   - Persist as `Move` row with `userId: null`
   - Update `Game.finalFen` and `Game.pgn`

5. Return `ComputerGameStateDto` (see mapping below).

**Category resolution helper:**
```typescript
function resolveCategory(seconds: number): TimeControlCategory {
  if (seconds < 180) return 'bullet';
  if (seconds < 600) return 'blitz';
  if (seconds < 1800) return 'rapid';
  return 'classical';
}
```
Check the actual `TimeControlCategory` enum values in Prisma schema first.

### `submitMove(gameId, userId, dto)` flow

1. Load game: `prisma.game.findUnique({ where: { id: gameId }, include: { moves: { orderBy: { ply: 'asc' } } } })`
2. Guard: game exists, `isVsComputer`, status is `'active'`, correct userId is set for the side to move.
3. Reconstruct engine state from DB:
   - Replay all stored moves to get current `Chess` position (or use `finalFen` if stored and non-null — but for correctness replay from starting FEN)
   - Actually: use `finalFen` if present (it's updated after each round), otherwise start from starting position
4. Validate user move via `EngineService.applyMove(state, { uci: dto.move }, Date.now())`
   - On `InvalidMoveError`: throw `BadRequestException`
5. Detect result after user move.
6. **If game not over:** call `stockfish.getBestMove(newFen, game.computerLevel)`, apply computer move, detect result again.
7. Persist in a transaction (`prisma.$transaction`):
   - Create `Move` for user move (`userId = userId`)
   - Create `Move` for computer move (`userId = null`) if computer moved
   - Update `Game`: `finalFen`, `pgn`, `status`, `result`, `resultReason`, `endedAt` if terminal
8. Return `ComputerGameStateDto`.

### `getGame(gameId, userId)` flow

1. Load game + moves.
2. Guard: game exists, `isVsComputer`, userId matches `whiteUserId` or `blackUserId`.
3. Return `ComputerGameStateDto`.

### `ComputerGameStateDto` mapping
```typescript
{
  gameId: game.id,
  fen: game.finalFen ?? STARTING_FEN,
  pgn: game.pgn ?? '',
  status: game.status,
  lastComputerMove: // last Move row where userId is null, or null
  result: game.result ?? null,
  resultReason: game.resultReason ?? null,
}
```

**`lastComputerMove`:** filter `moves` for those with `userId === null`, take last one's `uci`.

### PGN building
Use `EngineService.buildPgn(moves, headers)`. Reconstruct `EngineMove[]` from DB `Move[]`. Map:
```typescript
{ ply, san, uci, fenAfter: fenAfterMove, clockAfterMs: clockAfterMoveMs, moveTimeMs, by: ... }
```
The `by` field is `Color` — derive from ply: odd ply = white (`'w'`), even ply = black (`'b'`).

## Step 3: `ComputerGamesController`

**File:** `apps/api/src/computer-games/computer-games.controller.ts`

Pattern mirrors other controllers that use `@CurrentUser() user: { id: string }`.

```typescript
@UseGuards(SessionAuthGuard)
@Controller('computer-games')
export class ComputerGamesController {
  constructor(private readonly computerGames: ComputerGamesService) {}

  @Post()
  createGame(@CurrentUser() user: { id: string }, @Body() dto: CreateComputerGameDto) {
    return this.computerGames.createGame(user.id, dto);
  }

  @Post(':id/move')
  submitMove(
    @Param('id') id: string,
    @CurrentUser() user: { id: string },
    @Body() dto: ComputerMoveDto,
  ) {
    return this.computerGames.submitMove(id, user.id, dto);
  }

  @Get(':id')
  getGame(@Param('id') id: string, @CurrentUser() user: { id: string }) {
    return this.computerGames.getGame(id, user.id);
  }
}
```

No `@UsePipes(ValidationPipe)` needed at method level since DTOs are interfaces without class-validator decorators. The body is passed through directly — type safety is TypeScript compile-time only.

## Risks and Mitigations

| Risk | Likelihood | Mitigation |
|------|-----------|------------|
| `stockfish` npm package API differs from plan (CJS vs ESM, `addMessageListener` vs `onmessage`) | Medium | At implementation time, inspect `node_modules/stockfish/package.json` for `main`/`exports` fields and actual API surface before writing. Fall back to `onmessage` assignment if `addMessageListener` absent. |
| `stockfish` not installed (pnpm workspace symlink issue) | Low | Run `pnpm install` from repo root. If postinstall fails, check `pnpm-workspace.yaml` `allowBuilds.stockfish`. |
| `EngineService.applyMove` clock tick affects computer-vs-computer timing | Low | Computer games don't enforce clocks — pass `Date.now()` for `nowMs`; engine state clock is reconstructed each call so no drift accumulates. |
| `Move.userId` nullable migration breaks existing tests | Low | Existing tests mock Prisma — they won't hit the DB. Any test that asserts `userId` is non-null will still pass since real user moves still have `userId`. |
| `bestmove (none)` returned by Stockfish for terminal position | Medium | In `submitMove`, always call `detectResult` before calling Stockfish. Never call Stockfish if position is already terminal. |
| `TimeControlCategory` enum mismatch | Low | Read current enum values from schema before implementing `resolveCategory`. |
| Engine state reconstruction is expensive (replay all moves) | Low | For session 02, replay is fine. Future optimization: store serialized engine state. |

## Key Design Decisions Summary

| Decision | Rationale |
|----------|-----------|
| `Move.userId` → nullable | Computer moves have no DB user. Nullable FK is the honest representation. Migration is purely additive (DROP NOT NULL). |
| One Stockfish instance per `getBestMove()` call | No shared state needed between calls. Spawning per-call avoids race conditions and resource leaks in concurrent request handling. 1-second movetime keeps latency acceptable. |
| `isRated: false` for all computer games | ELO integrity — playing a bot shouldn't affect human rating. |
| Unrated game, `computerLevel` stored on Game | Supports future features (difficulty display, win-tracking by level) without a new table. |
| Replay moves from DB to reconstruct engine state | No serialized state persisted yet. Simple and correct for session 02; can be optimized later by storing `SerializableEngineState` on Game. |
| `prisma.$transaction` for move + game update | Atomicity — prevents partial writes where move is stored but game state is stale. |
| Computer moves stored in `Move` table with `userId = null` | Full game history preserved in the standard table. PGN, replay, analysis all work without special casing. |
