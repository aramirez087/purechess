# Session 02: Database Schema & Prisma

## Mission

Define the full Postgres schema for Purchess, wire up Prisma in the API, and ship the first migration. Every later session that touches the database will use the Prisma client generated here.

## Tasks

1. **Prisma schema (`apps/api/prisma/schema.prisma`)** modeling:
   - `User` — id, username (unique citext), email (unique citext), passwordHash (nullable for OAuth-only), avatarUrl, createdAt, updatedAt, lastLoginAt, isDisabled, isAdmin.
   - `OAuthAccount` — provider (google | apple), providerUserId, userId, createdAt. Composite unique on (provider, providerUserId).
   - `Session` — id, userId, ipAddress, userAgent, createdAt, expiresAt.
   - `PasswordResetToken` — id, userId, tokenHash, createdAt, expiresAt, usedAt.
   - `Rating` — id, userId, category (bullet | blitz | rapid), rating, ratingDeviation, volatility, gamesPlayed, updatedAt. Unique on (userId, category).
   - `RatingHistory` — id, userId, category, ratingBefore, ratingAfter, ratingDelta, gameId, createdAt.
   - `Game` — all fields from the PRD data model (whiteUserId, blackUserId, timeControlSeconds, incrementSeconds, category, isRated, result, resultReason, status, startingFen, finalFen, pgn, whiteRatingBefore, blackRatingBefore, whiteRatingAfter, blackRatingAfter, startedAt, endedAt, createdAt). Include `inviteToken` (nullable, unique) for friend invites.
   - `Move` — id, gameId, moveNumber, ply, userId, san, uci, fenAfterMove, clockAfterMoveMs, moveTimeMs, createdAt.
   - `Report` — id, reporterUserId, reportedUserId, gameId, reason, notes, status (open | reviewed | dismissed), createdAt, reviewedAt, reviewedByUserId.
   - `FairPlaySignal` — id, userId, gameId, signalType (low_variance_move_time, suspicious_accuracy, abnormal_streak, multi_account_ip, multi_account_fingerprint), score, payload (Json), createdAt.
   - `AdminAuditLog` — id, adminUserId, action, targetType, targetId, payload (Json), createdAt.
2. **Enums** in Prisma for: `GameResult`, `GameResultReason`, `GameStatus`, `TimeControlCategory`, `ReportStatus`, `OAuthProvider`, `FairPlaySignalType`.
3. **Indexes**:
   - `Game`: (whiteUserId, createdAt), (blackUserId, createdAt), (status, createdAt), (category, createdAt).
   - `Move`: (gameId, ply) unique.
   - `Rating`: (userId, category) unique.
   - `Session`: (userId), (expiresAt).
4. **Migration**:
   - `apps/api/prisma/migrations/0001_init/migration.sql` generated by `prisma migrate dev`.
   - Document the command in README.
5. **Prisma client wiring**:
   - `apps/api/src/database/prisma.service.ts` extends `PrismaClient` and implements `OnModuleInit` / `OnModuleDestroy` for clean shutdown.
   - `apps/api/src/database/database.module.ts` is `@Global()` and exports `PrismaService`.
6. **Shared types export**:
   - `packages/shared/src/db.ts` re-exports the Prisma enums (or pure TS enums if shared shouldn't import Prisma client). Choose the dependency-light approach: pure TS enums in shared, validated against Prisma schema in tests.
7. **Seed script** (`apps/api/prisma/seed.ts`):
   - 1 admin user (`admin@purchess.local` / known dev password).
   - 3 test users with usernames `alice`, `bob`, `carol`, all with rating 1500 across categories.
   - 2 completed games between them (one rated, one casual) with full move history and PGN.
8. **Verification**:
   - `pnpm --filter @purchess/api prisma migrate dev` runs cleanly.
   - `pnpm --filter @purchess/api prisma db seed` populates the test data.
   - `prisma generate` produces types usable by the API and shared package.
   - `pnpm typecheck` still passes.

## Deliverables

- `apps/api/prisma/schema.prisma` with the full model.
- Initial migration checked in.
- `PrismaService` and `DatabaseModule` wired in NestJS.
- `pnpm db:migrate`, `pnpm db:seed`, `pnpm db:reset` scripts in `apps/api/package.json`.
- Seed users/games ready for local dev.

## Notes for Downstream Sessions

- Always use the exported `PrismaService` — never instantiate `PrismaClient` directly.
- Use Prisma's enum types in code; do not redeclare them in TS.
- `Game.status` lifecycle: `pending` → `active` → `completed` | `aborted`. Add `invite_pending` for friend invites waiting on second player.
- Time fields: store everything as `Int` (seconds) or `BigInt` (ms) — pick one convention per field and stick to it. Use `Int` for seconds in Game, `Int` for ms in Move (Postgres int is 32-bit signed; max ~2.1e9 ms ≈ 24 days, plenty).
