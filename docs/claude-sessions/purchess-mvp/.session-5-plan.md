## Key Design Decisions

### 1. `GameResultReason` vs `GameTermination`

`packages/shared/src/game-result.ts` already exports `GameTermination` with all required values. Session spec names `GameResultReason`. Export alias from `chess.ts`:

```ts
export { GameTermination as GameResultReason } from './game-result.js';
```

### 2. `EngineState` — two forms

In-memory engine state contains a live `Chess` instance (not JSON-serializable). Strategy:

- `EngineState` (internal, API-layer only): has `position: Chess`; declared in `game-state.ts`, NOT in shared
- `SerializableEngineState` (shared, for Redis/wire): has `fen: string` instead of `position`

`toSerializable(state)` and `fromSerializable(s)` in `game-state.ts` convert between the two.

### 3. chess.js immutability pattern

`Chess` is mutable. Every engine function clones by FEN:

```ts
const chess = new Chess(state.position.fen());
```

Cheap (FEN parse is O(64)). No deep clone needed.

### 4. BigInt clock math

All internal clock arithmetic uses `BigInt` to prevent float drift:

```ts
const remaining = BigInt(clock.whiteMs) - BigInt(elapsedMs);
```

Serialized to `number` (ms) for wire/DB via `Number(remaining)`. BigInt never escapes engine layer.

### 5. Halfmove clock for 50-move rule

FEN field 5 is the halfmove clock. Parse directly:

```ts
export function halfmoveClock(fen: string): number {
  return parseInt(fen.split(' ')[4], 10);
}
```

No chess.js internal API needed.

### 6. Threefold repetition via fenHistory

`chess.js` tracks its own history only for moves played on that instance. Since engine uses fresh instances per move, we maintain `fenHistory` manually. Strip halfmove + fullmove fields before storing (position equality ignores move counts):

```ts
function positionKey(fen: string): string {
  return fen.split(' ').slice(0, 4).join(' ');
}
```

Threefold detected when `fenHistory.filter(f => f === positionKey(currentFen)).length >= 3`.

### 7. `EngineService` as thin NestJS wrapper

`@Injectable()` EngineService delegates to pure functions. No state stored in service — all state lives in `EngineState` objects owned by `GameService` (session 08). Tests exercise pure functions directly without NestJS.

### 8. Promotion validation

`validateMove` rejects pawn reaching rank 1/8 without valid promotion piece. UCI `e7e8q` encodes it; `from`/`to` object with `promotion: 'q'` also valid. Validator normalizes both.

### 9. `ClockSnapshot` internal type

Internal clock representation uses `bigint` fields to enforce BigInt discipline:

```ts
type ClockSnapshot = {
  whiteMs: bigint;
  blackMs: bigint;
  lastTickAt: bigint;
  incrementMs: bigint;
}
```

`EngineState.clock` uses this type. `serializeClock()` converts to `number` for wire/DB.

## Risks and Mitigations

| Risk | Mitigation |
|------|-----------|
| chess.js v1 API differences from training data | Install `chess.js@1.1.0`, read actual typedefs before coding |
| `BigInt` serialization via `JSON.stringify` throws | Convert to `number` in `toSerializable`; BigInt is internal only |
| Circular dep: shared `chess.ts` ↔ `game-result.ts` | `chess.ts` imports `from './game-result.js'` — one-way, no cycle |
| `GameTermination`/`GameResultReason` confusion | Both exported from shared; document alias in handoff |
| Test coverage < 90% | Explicit edge-case tests for castling, en passant pin, KvKB, KvKBB |
| `@types/chess.js` may be stale / bundled | Check if chess.js 1.x ships own types; use `@types/chess.js` only if needed |

### Step 4 — `clock.ts`

```ts
type ClockSnapshot = { whiteMs: bigint; blackMs: bigint; lastTickAt: bigint; incrementMs: bigint }

export function makeClock(whiteMs: number, blackMs: number, incrementMs: number, nowMs: number): ClockSnapshot
export function tickClock(clock: ClockSnapshot, nowMs: number, sideToMove: Color): ClockSnapshot
export function applyIncrement(clock: ClockSnapshot, side: Color): ClockSnapshot
export function isTimeout(clock: ClockSnapshot, nowMs: number, sideToMove: Color): boolean
export function serializeClock(clock: ClockSnapshot): { whiteMs: number; blackMs: number; lastTickAt: number; incrementMs: number }
```

### Step 5 — `result-detector.ts`

```ts
export function detectResult(
  chess: Chess,
  fenHistory: string[],
  clock: ClockSnapshot,
  nowMs: number
): { result: GameResult; reason: GameResultReason } | null
```

Check order:
1. Timeout (`isTimeout`)
2. Checkmate (`chess.isCheckmate()`)
3. Stalemate (`chess.isStalemate()`)
4. Insufficient material (`chess.isInsufficientMaterial()`)
5. Threefold repetition (count in `fenHistory`)
6. 50-move (`halfmoveClock(chess.fen()) >= 100`)

### Step 6 — `move-validator.ts`

```ts
export type MoveValidationResult =
  | { ok: true; newChess: Chess; san: string; uci: string }
  | { ok: false; reason: string }

export function validateMove(chess: Chess, intent: MoveIntent): MoveValidationResult
```

- Clone chess from FEN before move attempt
- Normalize intent: UCI `e7e8q` → `{ from: 'e7', to: 'e8', promotion: 'q' }`
- Reject pawn reaching back rank without promotion
- Return new `Chess` instance (rehydrated from result FEN) on success

### Step 7 — `game-state.ts`

```ts
export type EngineState = {
  gameId: string;
  whiteUserId: string | null;
  blackUserId: string | null;
  position: Chess;
  fenHistory: string[];          // position keys (first 4 FEN fields)
  moves: EngineMove[];
  pendingDrawOfferBy: Color | null;
  clock: ClockSnapshot;
  status: 'pending' | 'active' | 'completed' | 'aborted';
  result: GameResult | null;
  resultReason: GameResultReason | null;
}

export function createGame(opts: CreateGameOpts): EngineState
export function applyMove(state: EngineState, intent: MoveIntent, nowMs: number): EngineState  // throws InvalidMoveError
export function unmakeMove(state: EngineState): EngineState  // throws if no moves
export function toSerializable(state: EngineState): SerializableEngineState
export function fromSerializable(s: SerializableEngineState): EngineState
```

`applyMove` flow:
1. Call `tickClock(state.clock, nowMs, state.position.turn())` → check `isTimeout` → if yes, return completed state
2. `validateMove(state.position, intent)` → if `{ ok: false }`, throw `InvalidMoveError(reason)`
3. Compute `moveTimeMs = Number(BigInt(nowMs) - state.clock.lastTickAt)`
4. `applyIncrement(ticked_clock, side)` → `updateLastTickAt(clock, nowMs)`
5. Push `fenPosition(newChess.fen())` to `fenHistory`
6. Build `EngineMove` entry
7. `detectResult(newChess, fenHistory, newClock, nowMs)` → if result, return completed state
8. Return new `EngineState` (fully new object, no mutation)

`unmakeMove` uses `fenHistory` and `moves` array to restore prior position.

`InvalidMoveError` is a custom error class exported from `game-state.ts`.

### Step 8 — `pgn-builder.ts`

export interface PgnHeaders {

## Test Matrix

### fen-utils.spec.ts
- `startingFen()` returns known starting FEN string
- `fenPosition()` strips halfmove and fullmove fields (result has 4 space-separated parts)
- `halfmoveClock()` parses 0 from starting FEN, 3 from a FEN with clock=3
- FEN round-trip: `toFen(new Chess(knownFen)) === knownFen`

### clock.spec.ts
- `tickClock` decrements the moving side's time by elapsed ms
- `tickClock` clamps to 0 (never negative)
- `applyIncrement` adds increment to the side that just moved
- `isTimeout` returns true when time reaches 0
- `isTimeout` returns false when time > 0
- `serializeClock` converts bigint fields to number correctly

### result-detector.spec.ts
- Checkmate position (Scholar's mate final) → `{ result: WhiteWins, reason: Checkmate }`
- Stalemate position → `{ result: Draw, reason: Stalemate }`
- KvK → `{ result: Draw, reason: InsufficientMaterial }`
- KvKB → insufficient material
- KvKN → insufficient material
- KvKBB same color → insufficient (both bishops same color)
- Threefold: push same positionKey 3× to fenHistory → Draw/ThreefoldRepetition
- 50-move: FEN with halfmoveClock=100 → Draw/FiftyMoveRule
- Timeout: clock at 0 for active side → losing side loses
- Active game with no draw conditions → null

### move-validator.spec.ts
- Legal pawn push (e2e4) → `{ ok: true }`
- Illegal move (e2e5) → `{ ok: false }`
- Wrong turn (black pawn when white to move) → `{ ok: false }`
- Promotion missing piece (e7e8 uci without 5th char) → `{ ok: false }`
- Promotion with queen (e7e8q) → `{ ok: true }`
- Castling kingside when legal → `{ ok: true }`
- Castling through check → `{ ok: false }`
- En passant legal → `{ ok: true }`
- En passant when pinned (would expose king) → `{ ok: false }`
- Double check: only king moves valid; non-king move → `{ ok: false }`
- UCI and `from/to/promotion` object forms both accepted

### game-state.spec.ts
- `createGame` returns `status: 'pending'`, correct initial clock
- `applyMove` on pending game: status stays pending until explicitly activated (or: session defines when pending→active; check session spec — it says `createGame` returns state, activation is caller's concern; mark game active on first move)
- Scholar's mate 4-move sequence → `status: 'completed'`, `result: WhiteWins`, `reason: Checkmate`
- Invalid move throws `InvalidMoveError`
- `unmakeMove` after 1 move restores initial state exactly
- `toSerializable` → `JSON.stringify` → parse → `fromSerializable` → identical `fenHistory`, `moves`, `clock`
- Determinism: same move sequence from same start always yields same `fenHistory`

### pgn-builder.spec.ts
- Headers formatted as `[Tag "value"]` lines
- `Result` header present and matches result token at end
- Moves numbered correctly: `1. e4 e5 2. Nf3 Nc6 ...`
- Black's first move after incomplete first pair: `1... e5` (PGN standard)
- Empty move list → only headers + result token
- Round-trip: `new Chess().loadPgn(buildPgn(moves, headers))` → `chess.history()` matches original SAN list
