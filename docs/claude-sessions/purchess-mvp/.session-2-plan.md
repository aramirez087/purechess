## Key Design Decisions

### 1. citext for username/email

**Decision**: Use Prisma `postgresqlExtensions` preview feature + `@db.Citext` annotation.

```prisma
generator client {
  provider        = "prisma-client-js"
  previewFeatures = ["postgresqlExtensions"]
}

datasource db {
  provider   = "postgresql"
  url        = env("DATABASE_URL")
  extensions = [citext]
}
```

Fields `User.username` and `User.email` get `@db.Citext`.

**Rationale**: Case-insensitive uniqueness at DB level beats application-layer lowercasing. No runtime overhead. The extension is standard on Postgres/Neon.

**Risk**: citext extension must exist on the Postgres server. On Neon and stock Postgres it's available; `CREATE EXTENSION IF NOT EXISTS citext` is emitted by the migration automatically when extension is declared.

### 2. Enum alignment with shared package

The session spec names `TimeControlCategory` (bullet|blitz|rapid only — no classical for rated games). Shared package has `TimeCategory` with `Classical` and `Correspondence`. **Resolution**:
- Prisma enum `TimeControlCategory` = `bullet | blitz | rapid` (DB/rating categories only)
- `packages/shared/src/db.ts` re-declares the same values as pure TS const enums
- Shared `TimeCategory` (from `time-control.ts`) remains unchanged — it's a broader UI concept
- No conflict: different enum names, different use cases

`GameResult` in shared = `white_wins | black_wins | draw`. Prisma `GameResult` mirrors exactly.

`GameTermination` in shared maps to Prisma `GameResultReason`. Names differ intentionally: Prisma uses `GameResultReason` (session spec), shared uses `GameTermination` (chess domain term). `db.ts` exports `DbGameResultReason` to avoid collision with existing `GameTermination`.

### 3. Pure TS enums in `packages/shared/src/db.ts`

Shared package must not import `@prisma/client` (coupling violation). Strategy: duplicate enums as plain TS string enums in `db.ts`. Values must be **identical strings** to Prisma-generated values. A comment in `db.ts` warns that changes require Prisma schema sync.

### 4. `DATABASE_URL` vs `DATABASE_URL_DIRECT`

Neon (serverless Postgres) with PgBouncer requires two URLs:
- `DATABASE_URL` — pooled connection string (for runtime `PrismaClient`)
- `DATABASE_URL_DIRECT` — direct non-pooled string (for `prisma migrate dev` and `prisma db push`)

Prisma schema `datasource` block: `url = env("DATABASE_URL")`, `directUrl = env("DATABASE_URL_DIRECT")`.

For local dev, both can point to the same local postgres. `.env.example` gets both keys.

### 5. Seed script approach

- Use `bcryptjs` (pure JS, no native bindings) for password hashing
- Hardcode PGN for 2 games (Scholar's Mate variant + short draw) to avoid chess.js in seed
- Use `prisma.$transaction` for atomic seed
- Idempotent: check `findFirst` before creating admin to allow re-seeding

### 6. `PrismaService` implementation

```typescript
// Extends PrismaClient, NestJS lifecycle hooks for clean shutdown
export class PrismaService extends PrismaClient implements OnModuleInit, OnModuleDestroy {
  async onModuleInit() { await this.$connect(); }
  async onModuleDestroy() { await this.$disconnect(); }
}
```

### 7. Move.clockAfterMoveMs and moveTimeMs type

Session spec: `Int` for ms in Move. Prisma `Int` = 32-bit signed, max ~2.1B ms ≈ 24 days. Sufficient for any chess time control. Use `Int` (not `BigInt`) per spec note.

### 8. DatabaseModule global export

```typescript
@Global()
@Module({ providers: [PrismaService], exports: [PrismaService] })
export class DatabaseModule {}
```

Import once in `AppModule`. All other modules get `PrismaService` injected without re-importing.

## Risks and Mitigations

| Risk | Mitigation |
|------|------------|
| citext extension unavailable on local postgres | Migration SQL includes `CREATE EXTENSION IF NOT EXISTS citext`; Neon has it by default |
| `postgresqlExtensions` preview feature stability | It's been stable since Prisma 4.5; risk is low; if it fails fall back to application-layer lowercase + `@@map` |
| NestJS CJS import of `@prisma/client` | Prisma generates CJS-compatible output by default; no conflict |
| `ts-node` for seed on CJS project | Requires `--compiler-options '{"module":"CommonJS"}'`; documented in seed script |
| Enum value drift between Prisma and `db.ts` | Add comment warning; future sessions that change Prisma enums must update `db.ts`; note in handoff |
| `app.module.ts` outside touches glob | Single import line addition, necessary for DatabaseModule to function; documented as controlled exception in handoff |
| `shared/index.ts` outside touches glob | Single export line, same rationale |
| Migration requires live DB | Plan step 12 notes the requirement; if no local DB available, `prisma generate` still succeeds for typecheck; migration can be done last or against Neon dev branch |
