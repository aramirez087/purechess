## Order of Operations

Create the pnpm workspace and root tooling before scaffolding any app.

1. `pnpm-workspace.yaml` — declare `apps/*` and `packages/*`.

7. `prettier.config.js` + `.prettierignore`.
8. `eslint.config.js` — flat config with `@typescript-eslint/recommended`, no unused-vars, import rules.

### Phase 1: `packages/shared`
All types defined here; no logic, no side-effects.

**Files:**
```
packages/shared/
  package.json                  name: @purchess/shared
  tsconfig.json                 extends ../../tsconfig.base.json
  src/
    chess.ts                    Color, Piece, PieceType, Square, Move, GameState
    time-control.ts             TimeCategory enum, TimeControl interface, PRESETS map
    game-result.ts              GameResult enum, GameTermination enum
    rating.ts                   RatingCategory enum, RatingRecord interface
    ws-events.ts                WsEvent enum (all event names), payload types
    dto/
      auth.dto.ts               LoginDto, RegisterDto, AuthResponseDto
      game.dto.ts               CreateGameDto, MakeMoveDto, GameSummaryDto
      matchmaking.dto.ts        JoinQueueDto, QueueStatusDto
      user.dto.ts               UserProfileDto, UpdateProfileDto
    index.ts                    re-exports all
```

**Key design decisions:**
- `WsEvent` is a string enum so values are self-describing in logs.
- DTOs are plain interfaces (not classes) — NestJS `class-validator` decorators added in `apps/api` by wrapping them.
- `Square` = algebraic string `a1`–`h8` (not a number index) for readability across the stack.
- `GameState` includes `fen`, `pgn`, `turn`, `status`, `moves[]`, `clocks`.

## Key Design Decisions

| Decision | Rationale |
|---|---|
| `packages/shared` uses plain TS interfaces, not classes | Avoids NestJS decorator coupling in shared pkg; api wraps DTOs in classes locally |
| `Square` as `a1`–`h8` string | Human-readable in WS logs; chess.js compatible |
| `WsEvent` string enum | Values visible in DevTools without a lookup table |
| Flat ESLint config (v9) | Next.js 14 ships ESLint 9; flat config avoids dual-config confusion |
| `tsconfig.base.json` with `strict: true` | Downstream sessions can't soften it without touching base config |
| NestJS `@nestjs/terminus` for health | Standard; session 23 (observability) will extend it with DB/Redis checks |
| CORS: `http://localhost:3000` + env var `NEXT_PUBLIC_APP_URL` | Allows both local dev and deployed origin |

---

## Risks & Mitigations

| Risk | Mitigation |
|---|---|
| Next.js 14 App Router + TanStack Query v5 SSR hydration | Use `QueryClientProvider` only in a `'use client'` layout wrapper; don't fetch in server components yet (session 13 owns that) |
| pnpm hoisting conflicts between NestJS reflect-metadata and other packages | Pin `reflect-metadata` at root devDeps; ensure `emitDecoratorMetadata: true` only in api tsconfig |
| ESLint flat config incompatible with some legacy plugins | Use `@eslint/js`, `typescript-eslint` v7+ (supports flat config natively) |
| `packages/shared` not resolving in Next.js | Set `transpilePackages: ['@purchess/shared']` in `next.config.ts` |
| NestJS skeleton modules with empty controllers cause 404 on non-health routes | Expected — stubs return empty; downstream sessions fill them |
