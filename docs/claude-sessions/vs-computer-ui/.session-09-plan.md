## 1. `use-game-keyboard.ts` — Design

### Interface

```ts
export interface UseGameKeyboardOptions {
  /** Called when 'r' pressed and game is active. Should open confirm dialog, not resign directly. */
  onResign?: () => void;
  /** Called when 'n' pressed. Allowed even after game over. */
  onNewGame?: () => void;
  /** Called when 'u' pressed and game is active. */
  onTakeback?: () => void;
  /** Called when 'h' pressed and game is active. */
  onHint?: () => void;
  /** Called when 'f' pressed. Always allowed. */
  onFlip?: () => void;
  /** Arrow left — seek one ply back. */
  onSeekPrev?: () => void;
  /** Arrow right — seek one ply forward. */
  onSeekNext?: () => void;
  /** Home — seek to start. */
  onSeekStart?: () => void;
  /** End — seek to latest ply. */
  onSeekEnd?: () => void;
  /** When true, r/u/h are no-ops (but n/f and seek still work). */
  isGameOver?: boolean;
  /** When true, all keys are ignored. Use while page is loading. */
  disabled?: boolean;
}

export function useGameKeyboard(opts: UseGameKeyboardOptions): void
```

### Guard logic (in order)

1. `if (disabled) return` — loading state.
2. `const target = e.target as HTMLElement` → guard:
   `target instanceof HTMLInputElement || target instanceof HTMLTextAreaElement || target.isContentEditable`
   — mirrors `review-controls.tsx` pattern; adds `isContentEditable` for rich text fields.
3. Key dispatch table:

| Key | Condition | Action |
|-----|-----------|--------|
| `n` | always | `onNewGame?.()` |
| `r` | `!isGameOver` | `onResign?.()` |
| `u` | `!isGameOver` | `onTakeback?.()` |
| `h` | `!isGameOver` | `onHint?.()` |
| `f` | always | `onFlip?.()` |
| `ArrowLeft` | always | `e.preventDefault(); onSeekPrev?.()` |
| `ArrowRight` | always | `e.preventDefault(); onSeekNext?.()` |
| `Home` | always | `e.preventDefault(); onSeekStart?.()` |
| `End` | always | `e.preventDefault(); onSeekEnd?.()` |

4. `window.addEventListener('keydown', handleKey)` — single global listener.
5. Cleanup: `return () => window.removeEventListener('keydown', handleKey)`.
6. Dependency array: all callbacks + `isGameOver` + `disabled` (matches `review-controls.tsx` style).

### Rationale

- Pure hook (no internal state). Caller decides what each action does — clean separation.
- `'n'` allowed after game over so user can start a new game without clicking.
- Arrow keys + Home/End always available (ply scrub works during or after game).
- `e.preventDefault()` on arrow keys prevents browser scroll — same pattern as `review-controls.tsx`.

## 2. `live-announcer.tsx` — Design

### Interface

```ts
export interface LiveAnnouncerProps {
  /** SAN string of the last computer move, or null if none / human just moved. */
  lastComputerMoveSan: string | null;
  /** Human-readable result string ("You won", "You lost", "Draw") or null if game is active. */
  gameResult: string | null;
}

export function LiveAnnouncer({ lastComputerMoveSan, gameResult }: LiveAnnouncerProps): JSX.Element
```

### Rendering

```tsx
<div
  role="status"           // implies aria-live="polite" + aria-atomic="true"
  className="sr-only"     // visually hidden; Tailwind screen-reader utility
>
  {gameResult
    ? `Game over. ${gameResult}.`
    : lastComputerMoveSan
      ? `Computer played ${lastComputerMoveSan}.`
      : ''}
</div>
```

### Rationale

- `role="status"` = implicit `aria-live="polite"` per ARIA spec. Cleaner than bare `aria-live` attribute and gives testing-library a queryable role (`getByRole('status')`).
- `aria-atomic="true"` (implied by `role="status"`) ensures the full string is read, not just the changed character.
- `sr-only` keeps it out of the visual layout with no layout impact.
- Empty string when neither condition holds = no announcement (correct: no update to the live region DOM).
- Game result takes priority over last move — e.g. checkmate fires "Game over. You won." not "Computer played Qxh7."

### 3b. State additions (inside `ComputerGameClient`)

### 3c. Computed values (after `sanMoves` is defined)
```ts
const currentPly = seekPly ?? sanMoves.length;

// The last SAN move made by the computer (null when human just moved or no moves yet).
// White's moves are at 0-based even indices; black's at odd indices.
const lastComputerMoveSan: string | null = (() => {
  if (sanMoves.length === 0) return null;
  const lastIdx = sanMoves.length - 1;
  const lastMoveIsWhite = lastIdx % 2 === 0;
  return (game.computerColor === 'white') === lastMoveIsWhite ? sanMoves[lastIdx] : null;
})();
```

### 3d. Seek handlers
```ts
function handleSeek(ply: number) {
  setSeekPly(ply < sanMoves.length ? ply : null);
}
function handleSeekPrev() {
  setSeekPly((p) => {
    const from = p ?? sanMoves.length;
    return from > 1 ? from - 1 : 1;
  });
}
function handleSeekNext() {
  setSeekPly((p) => {
    if (p === null) return null;
    const next = p + 1;
    return next >= sanMoves.length ? null : next;
  });
}
function handleSeekStart() { setSeekPly(1); }
function handleSeekEnd()   { setSeekPly(null); }
```

### 3e. Wire `useGameKeyboard`
```ts
useGameKeyboard({
  onResign:    !isGameOver ? () => setConfirmResign(true) : undefined,
  onNewGame:   () => router.push('/play'),

  onTakeback:  undefined,   // not yet implemented; stub kept for future session
  onHint:      undefined,   // not yet implemented; stub kept for future session
```

## 4. Test Plan (`test/computer-game/a11y.test.tsx`)

Using `@testing-library/react` + `vitest` (already configured).

### Hook tests (`useGameKeyboard`)

```
pressing 'h' fires onHint
pressing 'u' fires onTakeback
pressing 'n' fires onNewGame even when isGameOver=true
pressing 'r' fires onResign when active, does nothing when isGameOver=true
pressing 'f' fires onFlip
ArrowLeft fires onSeekPrev
Home fires onSeekStart
pressing 'h' on an <input> element is ignored
pressing 'h' on a contenteditable element is ignored
pressing 'h' when disabled=true is ignored
listener is cleaned up on unmount (no calls after unmount)
```

**Key implementation note**: `renderHook` from `@testing-library/react`. Fire events with
`fireEvent.keyDown(window, { key: 'h' })` — the hook listens on `window` so dispatching to
`window` directly is correct. For the input-guard test, create an `<input>` in the DOM and call
`fireEvent.keyDown(inputEl, { key: 'h' })` — the event target is the input; the hook checks
`e.target instanceof HTMLInputElement` and returns early.

### Announcer tests (`LiveAnnouncer`)

```
renders aria status region
shows "Computer played Nf3." when lastComputerMoveSan="Nf3" and no result
updates text when lastComputerMoveSan changes from "Nf3" to "Qxh7"
shows "Game over. You won." when gameResult="You won"
shows empty string when both props are null
```

Query by `screen.getByRole('status')` — works because `role="status"` is explicit on the div.

## 6. Risks and Mitigations

| Risk | Mitigation |
|------|-----------|
| Global `keydown` listener conflicts with existing listeners | Hook attaches to `window`; review-controls.tsx also attaches to `window` — they coexist as independent handlers. Both guard against input targets. No conflict. |
| Double listener if hook mounts twice (React Strict Mode) | useEffect cleanup removes the listener before re-registration. Strict Mode double-invoke = mount, unmount, mount; final state = one listener. |
| `seekPly` desync with `sanMoves` (computer plays, sanMoves grows) | `currentPly = seekPly ?? sanMoves.length` auto-resets to live tail when `seekPly` is null. After computer move, `seekPly` stays at whatever the user set — if they're reviewing, they keep their position. |
| `lastComputerMoveSan` wrong when first move is computer's (black at idx 1) | Computation uses index-parity vs `game.computerColor`. If computer is black and last move index is odd → match → announce. Correct. |
| `role="status"` region fires on every render with same text | React only updates the DOM when text actually changes → no false announcements. |
| `scrollIntoView` in jsdom (MovePanel) | Already guarded with optional chaining in `move-panel.tsx` (`el?.scrollIntoView?.(…)`). No new risk. |
| `@radix-ui/react-alert-dialog` not in package.json | Check with `grep -r "alert-dialog" apps/web/package.json`. If absent, fall back to a simple `role="alertdialog"` div with focus trap via `useEffect`. |
