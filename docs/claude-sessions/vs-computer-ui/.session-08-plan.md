## Design Decisions & Rationale

### 1. PgnActions extended for FEN (not a new component)

`pgn-actions.tsx` currently has PGN copy + download. Adding an optional `fen?: string` prop
adds a "Copy FEN" button when present. This reuses existing infrastructure rather than forking.
`ReviewRail` passes both `pgn` and `fen` from `ComputerGameStateDto`.

### 2. ReviewRail placement in computer-game-client

Place `<ReviewRail pgn={game.pgn} fen={game.fen} gameId={gameId} />` inside the rightRail
`div`, between the `MovePanel GameRail` and the `BoardControlBar`. It shows at all times
(active and completed games), giving players the ability to copy/share a position mid-game.

### 3. isVsComputer as non-optional in shared DTO

`GameHistorySummaryDto.isVsComputer: boolean` (not optional). All existing `Game` rows have
`isVsComputer = false` (schema default), so the API can always populate it. Making it optional
would require null-guards everywhere in the UI.

### 4. opponentUsername fix for computer games in API

Current `fetchGames` sets `opponentUsername = g.blackPlayer?.username ?? ''` which gives `''`
for computer games (the computer-side player is null). Fix: check `g.isVsComputer` first and
return `'Computer'` (or `'Level N'` using `g.computerLevel`). The Prisma query already
selects `whitePlayer`/`blackPlayer`; just add `isVsComputer` and `computerLevel` to the
`findMany` select/fields — they're direct columns, no relation join needed.

### 5. vsComputer URL param naming

Use `vsComputer=true` in the URL (simpler than `isVsComputer=true`). The JS variable stays
`isVsComputer` to match the DTO field. The filter shows only computer games when active;
clearing it shows all games (the API returns all when `isVsComputer` is absent).

### 6. Deep-link review: web-only fallback approach (no new API endpoint)

`getReview` already hits `GET /games/:id`. For a computer-game ID, that endpoint returns 404.
Rather than adding a new API endpoint, extend `getReview` to:
  1. Try multiplayer endpoint (`GET /games/:id`).
  2. On 404, try computer-game state (`GET /api/computer-games/:id`).
  3. Only proceed if `status === 'completed'` (no review for active games).
  4. Reconstruct `GameReview` via chess.js PGN replay (server-side — chess.js is ESM-safe
     in Next.js server components since it's already used client-side).

This avoids any API changes and keeps the review fully client-reconstructed.

### 7. Missing timeControl in ComputerGameStateDto

`ComputerGameStateDto` does not expose `timeControlSeconds`. For the review page, use a
hardcoded fallback:
```typescript
{ initialSeconds: 0, incrementSeconds: 0, category: TimeCategory.Rapid, label: 'vs Computer' }
```
The `ReviewClient` shows this in an `InfoRow`; a generic label is acceptable for v1. Adding
`timeControlSeconds` to the DTO is future work.

### 8. currentUser threading into getReview

`games/[gameId]/page.tsx` already fetches `currentUser`. Update `getReview` signature to
`getReview(gameId: string, currentUser?: { id: string; username: string } | null)`. The
computer-game review uses `currentUser.username` for the human player's display name; falls
back to `'Player'` if unauthenticated.

### 9. Result/Termination mapping

`ComputerGameStateDto.result` is `'white_wins' | 'black_wins' | 'draw' | null`. Map to
`GameResult` enum. `ComputerGameStateDto.resultReason` maps to `GameTermination`; use a
`REASON_MAP` constant (same keys as `REASON_LABELS` in `computer-game-client.tsx`).

## Risks & Mitigations

| Risk | Severity | Mitigation |
|------|----------|-----------|
| chess.js `loadPgn` fails on malformed PGN | Low | Wrap in try/catch; return `[]` moves; the review will render an empty move list rather than crashing. |
| `isVsComputer` non-optional breaks TS if API misses it | Medium | Add field to `fetchGames` return object in the same commit as the DTO change; build-check before moving on. |
| `VirtualizedBody` in `game-history-list.tsx` wraps `GameHistoryRow` (which returns `<tr>`) inside another `<tr>` — nested `<tr>` invalid HTML | Low/Pre-existing | Do NOT fix; note it; badge changes only touch `GameHistoryRow` internals. |
| `GameReview.result` is required but computer game may still be active | Low | Return `null` from `getReview` for non-completed computer games — same as if game not found. Page renders 404. |
| `generateMetadata` calls `getReview` without `currentUser` | Low | `currentUser` is optional; fallback to `'Player'` and `'Computer'` for metadata title. |
| `@purechess/shared` build must run before web typecheck | — | Always run `pnpm --filter @purechess/shared build` first in CI and locally. |

### Phase 6: Deep-link review

**Step 12** — `apps/web/src/services/game-review.service.ts`

Add helper `buildReviewFromComputerGame`:
```typescript
import { Chess } from 'chess.js';
import type { ComputerGameStateDto } from '@purechess/shared';
import { GameResult, GameTermination, TimeCategory } from '@purechess/shared';
import type { GameReview } from '@/types/game-review';
import type { WireMove } from '@purechess/shared';

const RESULT_MAP: Record<string, GameResult> = {
  white_wins: GameResult.WhiteWins,
  black_wins: GameResult.BlackWins,
  draw: GameResult.Draw,
};

const TERM_MAP: Record<string, GameTermination> = {
  checkmate: GameTermination.Checkmate,
  resignation: GameTermination.Resignation,
  timeout: GameTermination.Timeout,
  stalemate: GameTermination.Stalemate,
  insufficient_material: GameTermination.InsufficientMaterial,
  threefold_repetition: GameTermination.ThreefoldRepetition,
  fifty_move_rule: GameTermination.FiftyMoveRule,
  draw_agreement: GameTermination.DrawAgreement,
  abandonment: GameTermination.Abandonment,
};

function buildMovesFromPgn(pgn: string): WireMove[] {
  if (!pgn.trim()) return [];
  try {
    const chess = new Chess();
    chess.loadPgn(pgn);
    const history = chess.history({ verbose: true });
    const replay = new Chess();
    return history.map((m, i) => {
      replay.move(m.san);
      return {
        ply: i + 1,
        san: m.san,
        uci: m.from + m.to + (m.promotion ?? ''),
        fenAfter: replay.fen(),
        clockAfterMs: 0,
        moveTimeMs: 0,
        by: m.color,
      };
    });
  } catch {
    return [];
  }
}

function buildReviewFromComputerGame(
  state: ComputerGameStateDto,
  currentUser: { id: string; username: string } | null,
): GameReview {
  const humanColor = state.computerColor === 'white' ? 'black' : 'white';
  const human = {
    id: currentUser?.id ?? 'player',
    username: currentUser?.username ?? 'Player',
    rating: 0,
  };
  const computer = { id: 'computer', username: 'Computer', rating: state.computerLevel };

  const white = state.computerColor === 'white' ? computer : human;
  const black = state.computerColor === 'black' ? computer : human;

  return {
    id: state.gameId,
    white,
    black,
    moves: buildMovesFromPgn(state.pgn),
    finalFen: state.fen,
    pgn: state.pgn,
    result: RESULT_MAP[state.result ?? ''] ?? GameResult.Draw,
    termination: TERM_MAP[state.resultReason ?? ''] ?? GameTermination.Resignation,
    timeControl: {
      initialSeconds: 0,
      incrementSeconds: 0,
      category: TimeCategory.Rapid,
      label: 'vs Computer',
    },
    rated: false,
    startedAt: new Date(0).toISOString(),
  };
}
```

Add `getComputerGameReview`:
```typescript
async function getComputerGameReview(
  gameId: string,
  currentUser: { id: string; username: string } | null,
): Promise<GameReview | null> {
  const apiUrl = process.env.API_INTERNAL_URL ?? 'http://localhost:4000';
  try {
    const res = await fetch(`${apiUrl}/api/computer-games/${gameId}`, { cache: 'no-store' });
    if (!res.ok) return null;
    const state = (await res.json()) as ComputerGameStateDto;
    if (state.status !== 'completed') return null;
    return buildReviewFromComputerGame(state, currentUser);
  } catch {
    return null;
  }
}
```
