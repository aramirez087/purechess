## 2. Design Decisions

### 2.1 `getEngineBackend()` returns `null` when unset

**Current behavior:**  
`getEngineBackend()` always returns `'ts'` as its final fallback. This means `ENGINE_BACKEND=native` in fly.toml is the only path to native mode — the code itself never defaults to native.

**Change:**  
Return `null` when neither `ENGINE_SHADOW` nor `ENGINE_BACKEND` is explicitly set. Callers apply the environment-aware default:

```ts
export function getEngineBackend(): 'native' | 'ts' | 'shadow-ts' | null {
  if (process.env.ENGINE_SHADOW === '1') return nativeAvailable ? 'shadow-ts' : 'ts';
  const env = process.env.ENGINE_BACKEND;
  if (env === 'native' && nativeAvailable) return 'native';
  if (env === 'ts') return 'ts';
  return null;  // no explicit override — let caller decide
}
```

**Rationale:** Belt-and-suspenders. fly.toml `[env]` sets `ENGINE_BACKEND=native` for prod (task 4), so in practice `getEngineBackend()` returns `'native'` without hitting the null path. But if a deploy accidentally drops the env var (fly.toml merge conflict, manual `fly secrets unset`), the production fallback in the factory ensures we stay on native. The ts default remains for local dev where `NODE_ENV !== 'production'`.

### 2.2 `chess.module.ts` factory uses `??` prod-default + Sentry breadcrumb

```ts
useFactory: (backendHint: string | null) => {
  const backend = backendHint ?? (process.env.NODE_ENV === 'production' ? 'native' : 'ts');

  let adapter: EngineAdapter;
  if (process.env.ENGINE_SHADOW === '1' && nativeAvailable) {
    adapter = new ShadowAdapter(new TsEngineAdapter(), new NativeEngineAdapter());
  } else if (backend === 'native' && nativeAvailable) {
    adapter = new NativeEngineAdapter();
  } else {
    adapter = new TsEngineAdapter();
  }

  try {
    Sentry.addBreadcrumb({ category: 'engine', message: `engine booted: ${adapter.name()}` });
  } catch {
    // Sentry not initialized (dev/test) — silently ignore
  }

  return adapter;
},
```

**Note:** `ENGINE_BACKEND` token provide changes from `process.env.ENGINE_BACKEND ?? 'ts'` to `getEngineBackend()` (which can now be `null`). The token type changes to `string | null`. No other consumers of `ENGINE_BACKEND` token — only `EngineService` injects `ENGINE_ADAPTER`.

**Sentry import:** `import { Sentry } from '../observability/sentry';` — already exported from `sentry.ts:16`.

### 2.3 `fly.toml` env var is source of truth for prod

`ENGINE_BACKEND = "native"` under `[env]` in `apps/api/fly.toml` makes the value:
- Visible in PR review (no secrets required for this non-sensitive config)
- Reproducible across `fly deploy` invocations
- Rollback-able by removing/changing the line + `fly deploy`

**Rollback path** (keep TS adapter): remove/comment the line and `fly deploy`, OR `fly secrets set ENGINE_BACKEND=ts -a purechess-api` (secrets override fly.toml env). Documented in the runbook.

### 2.4 CI `engine-shadow` job — no change needed

The existing `ci.yml` already triggers on `push: branches: [main]`. The `engine-shadow` job depends on `test` and runs on all CI triggers. Session requirement ("runs on every push to main") is already met. Document this in the handoff.

### 2.5 Native binary must be present in the Docker image

`nativeAvailable` is detected at module load time via a `try { require('@purechess/engine-native') }`. If the binary is absent, the factory falls back to `TsEngineAdapter` silently (existing behavior from WP4/WP5). The Sentry breadcrumb will show `engine booted: ts` in that case — useful signal.

The Dockerfile (multi-stage, already set up in WP3) must copy the compiled `.node` binary into the final image. Confirm this is in place before staging deploy — read `apps/api/Dockerfile` during the implementation session and verify the `COPY --from=rust-builder` stage. If missing, that is WP3 scope debt that must be resolved before cutover.

## 5. Runbook Content Spec (`docs/runbooks/engine-cutover.md`)

### Pre-cutover checklist
- [ ] `pnpm engine:shadow` exits 0 on latest main (ts-vs-ts, 203 traces)
- [ ] Sentry: zero `engine.shadow.divergence` events in past 7 days
- [ ] Staging deployed with `ENGINE_BACKEND=native`, Sentry breadcrumb confirmed

- [ ] On-call engineer available for 48h post-cutover
- [ ] Rollback command tested on staging (see below)

### Cutover steps
```bash
# Prod deploy (ENGINE_BACKEND=native is in fly.toml)
fly deploy -a purechess-api
curl https://purechess-api.fly.dev/api/health
# Verify Sentry breadcrumb: "engine booted: native"
```

### Monitoring
- **Sentry:** Filter by tag `engine.adapter=native`. Watch error rate on `validateMove`, `legalMoves`, `detectResult`.
- **Metrics endpoint:** `GET /api/metrics` — `moveValidations`, `engineErrors` counters.
- **Divergence counter:** Sentry events tagged `engine.adapter=shadow-ts` (should be 0 — shadow mode not active in prod at this point).
- **Game completion rate:** Games table in DB — ratio of `completed` to `active+completed` games over rolling 1h window.

### Rollback thresholds (ANY triggers immediate rollback)
- Any FEN divergence between engine output and expected board state in a live game
- Error rate on `validateMove` > 0.5% sustained for 5 minutes
- p99 `validateMove` latency > 50ms (was ~2ms in shadow mode testing)
- Single SEV-1 game-state bug (wrong legal moves, illegal move accepted, wrong result detected)

### Rollback command
```bash
# Override via secrets (takes precedence over fly.toml env)
fly secrets set ENGINE_BACKEND=ts -a purechess-api
fly deploy -a purechess-api
# Verify: Sentry breadcrumb "engine booted: ts"
```

The `.node` binary stays in the image — rollback is one env var change + redeploy (~90s).

## 7. Known Constraints / Gotchas

- **`engine/index.ts` does NOT call `getEngineBackend()`** — it reads `process.env.ENGINE_BACKEND` directly. This file creates a module-level singleton `engine` used by scripts. No change needed there; the NestJS DI path (chess.module.ts) is what matters for production.
- **`nativeAvailable` is module-level state** in `engine-backend.config.ts`. It's evaluated once at module load. If the binary isn't present at startup, the factory gracefully falls back to TS regardless of `ENGINE_BACKEND` env var. Breadcrumb confirms which adapter is actually running.
- **Do NOT use `fly secrets` for `ENGINE_BACKEND`** — keep it in fly.toml so it's PR-reviewable. Use secrets only for the rollback override (`fly secrets set ENGINE_BACKEND=ts` during an incident), which takes precedence.
- **`@typescript-eslint/no-var-requires`** — `chess.module.ts` has no `require()` calls; Sentry import is ESM-style. No lint suppression needed.
- **Staging DB:** Must be a separate database from prod. Do not point staging at the prod Neon DB.
- **48h observation before writing GO verdict** — the handoff go/no-go note is the last thing written, after watching metrics.
